# 主ラベル4値による話したい度推定システム設計メモ

## 1. この設計メモの目的

この文書は、Codex に実装方針を正確に伝えるための設計メモである。  
本システムでは、ユーザの発話内容を LLM で **主ラベル4値** に分類し、その分類結果を観測としてベイズ更新を行うことで、ユーザの潜在的な **話したい度** を逐次推定する。

ここで重要なのは、現段階では **次にどのような発話を返すかの最適化** ではなく、あくまで **話したい度を安定して推定すること** を主目的とする点である。

そのため、分類カテゴリは必要以上に細かくせず、ベイズ推定に用いやすい4つの主ラベルに限定する。  
尤度や事前確率の具体的な値はこの文書では固定しない。  
それらは Codex 側で実装時に設計・調整するものとする。

---

## 2. システムの基本方針

本システムは、各ターンにおいて以下の流れで動作する。

1. ユーザ発話を取得する  
2. その発話内容を LLM により主ラベル4値のいずれかに分類する  
3. その主ラベルを観測値としてベイズ更新を行う  
4. 更新後の確率を、その時点のユーザの「話したい度」とみなす

ここでいう「話したい度」は、ユーザがその時点でどの程度会話に関与したいか、どの程度話を続ける方向にあるかを表す潜在状態である。  
この値は直接観測できないため、発話内容から推定する。

---

## 3. 主ラベル4値の考え方

主ラベルは、ユーザ発話が会話継続の観点からどの程度前向きかを表す。  
1発話につき、**必ず1つだけ** 主ラベルを付与する。

採用する主ラベルは以下の4つである。

- `ACTIVE`
- `RESPONSIVE`
- `MINIMAL`
- `DISENGAGE`

この4つは、会話参加の強さの連続体として以下のように考える。

`DISENGAGE -> MINIMAL -> RESPONSIVE -> ACTIVE`

左ほど話したい度が低い方向、右ほど話したい度が高い方向を表す。

---

## 4. 各主ラベルの定義

### 4.1 ACTIVE

#### 定義
ユーザが受け身に答えるだけではなく、**自発的に話を広げている**、または **会話を続けようとする積極性が明確に見られる** 発話。

#### 特徴
- 質問に対して必要最小限を超えて詳しく答えている
- 自分から追加情報を出している
- 話題を拡張している
- 相手に聞き返すなど、双方向性が高い
- 会話を続けたい姿勢が読み取れる

#### 典型例
- 「高校の頃によくその公園に行ってたんですよ。特に夏祭りが好きで、毎年楽しみにしていました」
- 「それで思い出したんですが、その近くに駄菓子屋もあったんです」
- 「逆に、あなたはそういう場所に行ったことありますか？」

#### 判定のポイント
- 単に情報量が多いだけでなく、**会話を前に進める方向** に働いていること
- 自発性、拡張性、継続意欲が見えること

---

### 4.2 RESPONSIVE

#### 定義
ユーザが質問や話題に対して **きちんと応答している** が、そこまで強い自発的拡張は見られない発話。

#### 特徴
- 問いに対して意味のある返答をしている
- 内容はあるが、必要以上には広げない
- 会話を拒否してはいない
- 会話には参加しているが、主導的ではない

#### 典型例
- 「はい、行ったことがあります」
- 「学生の頃です」
- 「兄と一緒でした」
- 「好きでした」

#### 判定のポイント
- 応答としては十分成立している
- ただし、自発的な話題展開や詳細な拡張までは見られない

---

### 4.3 MINIMAL

#### 定義
ユーザが **最小限の反応はしている** が、会話を広げる意志は弱く、内容的にも乏しい発話。

#### 特徴
- 相づちに近い
- 短い返答のみ
- 曖昧で広がりにくい
- 受け身で消極的
- 即座の拒否ではないが、積極性は低い

#### 典型例
- 「うん」
- 「そうですね」
- 「へえ」
- 「まあ」
- 「たぶん」
- 「別に」
- 「そうかも」

#### 判定のポイント
- 会話を完全に切ってはいない
- しかし、この発話単体からは会話継続への強い意欲は読み取れない
- 短さだけでなく、**内容の広がりの乏しさ** も重要

---

### 4.4 DISENGAGE

#### 定義
ユーザが **会話継続に消極的** であることが明確、または **終了・拒否・回避** の方向性が強く表れている発話。

#### 特徴
- 会話を終えたい意思がある
- その話題を避けたい意思がある
- 明確な拒否や不満が含まれる
- 会話参加意欲の低下が強く示される

#### 典型例
- 「もういいです」
- 「その話はしたくないです」
- 「今日はこのへんで」
- 「別に話したくないです」
- 「違う話にしてください」

#### 判定のポイント
- MINIMAL と異なり、**会話から離れようとする方向性** が明確であること
- 単なる短い返答ではなく、拒否・終了・回避の意味を伴うこと

---

## 5. ラベル判定の基本原則

### 5.1 情報量よりも「会話継続への向き」を優先する
分類では、発話の長さや語数だけでなく、  
**その発話が会話を続ける方向に働いているか** を重視すること。

例えば、短くても明確に質問へ答えていれば `RESPONSIVE` になりうる。  
逆に、やや長くても拒否や終了意図が強ければ `DISENGAGE` とする。

---

### 5.2 感情の強さではなく、会話参加姿勢をみる
感情表現があるかどうかよりも、その発話が  
- 続けたい  
- とりあえず応じている  
- 最小限しか応じていない  
- 離れたい  
のどれに近いかで判断する。

---

### 5.3 必ず4値のどれか1つに落とす
曖昧なケースでも、最終的には必ず1つの主ラベルを出力する。  
複数ラベル同時付与は行わない。

---

## 6. 境界事例の扱い

### 6.1 `MINIMAL` と `RESPONSIVE` の違い
#### MINIMAL
- 「うん」
- 「そうですね」
- 「まあ…」

→ 内容が薄く、広がりがない。質問への実質的回答になっていない、または極めて弱い。

#### RESPONSIVE
- 「はい、好きでした」
- 「中学生の頃です」
- 「家族と行きました」

→ 短くても、質問に対する情報として成立している。

**判断基準**  
短さではなく、「質問への意味ある返答として成立しているか」で分ける。

---

### 6.2 `RESPONSIVE` と `ACTIVE` の違い
#### RESPONSIVE
- 「はい、行ったことがあります」
- 「友達と行きました」

#### ACTIVE
- 「はい、行ったことがあります。特に春になると桜がきれいで、友達とよく写真を撮っていました」

**判断基準**  
必要最小限を超えて、自発的に情報を追加し、話題を広げているなら `ACTIVE`。

---

### 6.3 `MINIMAL` と `DISENGAGE` の違い
#### MINIMAL
- 「うーん」
- 「別に」
- 「そうかも」

#### DISENGAGE
- 「もういいです」
- 「話したくないです」
- 「その話はやめたいです」

**判断基準**  
短くても、拒否・終了・回避の方向性が明確なら `DISENGAGE`。  
単に消極的なだけなら `MINIMAL`。

---

## 7. LLM分類器に求める出力

LLM は、各ユーザ発話に対して以下のような構造化結果を返すことが望ましい。

```json
{
  "primary_label": "ACTIVE | RESPONSIVE | MINIMAL | DISENGAGE",
  "reason": "判定理由を簡潔に記述"
}
```

### 出力要件
- `primary_label` は必ず4値のいずれか
- `reason` には、なぜそのラベルにしたかを1〜3文程度で簡潔に書く
- 不要に長い説明は不要
- 実装時には `reason` をログ保存してもよいが、ベイズ更新で直接使うのは `primary_label` のみでよい

---

## 8. ベイズ推定との接続

LLM が出力した `primary_label` は、そのままベイズ更新の観測カテゴリとして用いる。

例:
- `ACTIVE` は、話したい度が高いことを支持する観測
- `RESPONSIVE` は、中程度に話したい度を支持する観測
- `MINIMAL` は、話したい度がやや低いことを示す観測
- `DISENGAGE` は、話したい度が低いことを強く示す観測

ただし、具体的にどのラベルにどの尤度を割り当てるかは、この文書では固定しない。  
その値は実装段階で Codex 側が設計するものとする。

本設計メモでは、**4つの主ラベルを観測カテゴリとしてベイズ更新を行う** という構造のみを定義する。

---

## 9. 実装時の重要な注意点

### 9.1 まずは主ラベル4値の安定性を優先する
将来的に補助ラベルやより細かい分類を追加する可能性はあるが、現段階では不要。  
まずは主ラベル4値の定義を明確にし、LLM が安定して分類できることを重視する。

---

### 9.2 ラベル定義の一貫性を保つ
LLM プロンプトでは、各ラベルの定義・典型例・境界例を明示し、分類基準がぶれないようにする。  
特に `MINIMAL` と `RESPONSIVE`、`RESPONSIVE` と `ACTIVE`、`MINIMAL` と `DISENGAGE` の区別を丁寧に指示すること。

---

### 9.3 長さだけで判定しない
短い発話でも `RESPONSIVE` になりうる。  
長い発話でも、拒否や終了意図が中心なら `DISENGAGE` になりうる。  
したがって、表面的な文字数ではなく、会話参加姿勢に基づいて判断すること。

---

## 10. Codex への実装依頼の要旨

Codex には以下の方針で実装してもらいたい。

- ユーザ発話を入力として受け取る
- LLM を用いて、その発話を `ACTIVE / RESPONSIVE / MINIMAL / DISENGAGE` の4値に分類する
- 分類理由も構造化して返せるようにする
- 分類結果の主ラベルを観測としてベイズ更新を行う
- ベイズ更新により、潜在状態としての「話したい度」を逐次推定する
- 尤度や事前確率の具体値は、Codex 側で設計・調整可能にする
- 将来的な拡張は可能だが、現段階では主ラベル4値のみを用いる

---

## 11. 最終的な設計意図

本設計の狙いは、ユーザ発話を過度に複雑に分類することではない。  
目的は、**会話参加の強さを表す少数の安定したカテゴリを用いて、話したい度を推定可能な形に落とし込むこと** である。

そのため、
- ラベル数は4つに絞る
- 1発話に1ラベルのみ付与する
- そのラベルをベイズ更新の観測として使う
という構造を採用する。

これにより、分類器の安定性とベイズ推定の扱いやすさを両立する。
